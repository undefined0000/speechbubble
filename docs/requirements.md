# 吹き出し自動挿入ツール 要件定義（初版）

- 作成日: 2026-03-04
- 対象リポジトリ: `speechbubble`
- 目的: キャラクターが写ったイラストから、自然な吹き出しの「形状」「位置」「しっぽ向き」を自動決定し、吹き出し入り画像を出力する

## 1. 背景とゴール

同人誌制作では、吹き出し配置に時間がかかり、ページ全体の読み順や視線誘導の調整が手作業になりやすい。  
本ツールは、以下を自動化して作業時間を短縮する。

- 顔検出（誰が話しているかの候補抽出）
- 吹き出し配置候補の生成
- 画像内オブジェクトとの衝突回避
- 自然な吹き出し形状としっぽ方向の選択
- 最終画像の書き出し（将来はメタデータ出力も）

## 2. スコープ定義

### 2.1 MVP（初期スコープ）

- 単一画像入力（PNG/JPEG）
- 横書き日本語テキストの吹き出し挿入
- 1画像あたり 1〜4 吹き出し
- 顔位置に紐づくしっぽ付き吹き出し（標準形: 楕円 / 角丸）
- CLI実行 + 簡易Web UI（ローカル）
- 出力: 合成済み画像 + 位置情報JSON

### 2.2 初期スコープ外（後続）

- 縦書き本文の本格レイアウト
- コマ割り/ページ全体の読み順最適化
- 効果音や長文組版の自動整形
- 複数ページ一括での全体最適化
- 漫画特化の学習済み独自モデルの本格運用

## 3. ユースケース

- U1: イラスト1枚とセリフを入れると、自然な吹き出し入り画像を自動生成
- U2: 自動配置後、ユーザーが微調整して再出力
- U3: 1キャラ/複数キャラ画像で、話者ごとに吹き出しを分離
- U4: バッチ処理で複数枚を連続処理

## 4. 機能要件

### 4.1 入出力

- FR-001: 入力画像を受け取り、内部処理用に正規化する
- FR-002: 吹き出しテキスト（複数可）を受け取り、文字領域サイズを算出する
- FR-003: 合成結果画像を保存する
- FR-004: 吹き出しバウンディングボックス、しっぽ先端、スコアをJSON保存する

### 4.2 顔・話者推定

- FR-010: 顔検出を行い、顔矩形を得る
- FR-011: 顔ランドマークから「しっぽ接続点」（口元/顎/顔中心）を推定する
- FR-012: 複数顔時にセリフと話者を紐付ける（MVPはユーザー指定優先）

### 4.3 吹き出し候補生成

- FR-020: テキストサイズから必要吹き出し外接矩形を算出する
- FR-021: 顔周囲8方向（上/右上/右/右下/下/左下/左/左上）に候補配置を生成
- FR-022: 画像境界外にはみ出さない候補へ補正する
- FR-023: 候補同士の重なりを解消する

### 4.4 自然さスコアリング

- FR-030: 以下コストの重み付き合計で候補を評価し、最小コスト解を採用する
- FR-031: 顔・主要被写体への重なりペナルティ
- FR-032: 画像の高サリエンシー領域への重なりペナルティ
- FR-033: 吹き出し間重なりペナルティ
- FR-034: しっぽ長過大/不自然角度ペナルティ
- FR-035: 読み順違反（右上→左下/上→下の規則破り）ペナルティ

### 4.5 描画

- FR-040: 吹き出し本体（楕円/角丸/雲形）を描画
- FR-041: しっぽを顔アンカーへ接続
- FR-042: テキストを余白・行間・禁則を考慮して描画
- FR-043: 輪郭線・塗り・透過を指定可能にする

### 4.6 操作性

- FR-050: 自動結果に対する手動移動・形状変更・再最適化を提供
- FR-051: 処理ログ（検出数/候補数/採択理由）を保存

## 5. 非機能要件

- NFR-001: 2048px級画像1枚を CPU で 2〜5秒以内（MVP目標）
- NFR-002: 同入力に対して再現可能な結果（seed固定）
- NFR-003: 失敗時にフォールバック（中央上配置など）で必ず出力
- NFR-004: 依存ライブラリのライセンスを明示
- NFR-005: Windows環境で動作（将来 Linux/macOS）

## 6. 品質評価指標（受け入れ基準）

- Q-001: 顔検出再現率（検証画像での検出率）
- Q-002: 主要被写体重なり率（吹き出し領域と主要被写体マスクの交差率）
- Q-003: しっぽ接続妥当率（手動評価）
- Q-004: 編集削減率（手作業配置に対する時間短縮）
- Q-005: 主観自然度（5段階評価、平均4.0以上を目標）

## 7. 技術選定（比較）

### 7.1 方式候補

- A. ルールベース中心（CV + 幾何最適化）
- B. 学習ベース中心（配置を直接予測）
- C. ハイブリッド（検出は学習、配置はルール最適化）

### 7.2 推奨方式

推奨: **C. ハイブリッド**  
理由:

- 初期成功率が高い（学習データが少なくても成立）
- 結果の説明可能性が高い（なぜその位置かを説明しやすい）
- 将来、学習モデルを段階的に差し替え可能

## 8. 推奨技術スタック

### 8.1 言語/基盤

- Python 3.11 以上（Windows優先）
- 依存管理: `uv` + `pyproject.toml`

### 8.2 画像処理/推論

- OpenCV: 前処理・図形描画・補助CV処理
- MediaPipe Tasks: 顔検出/ランドマーク抽出
- Pillow: 日本語テキスト描画（`libraqm`有効化）
- （任意）PyTorch: 高精度マスク推定を行う場合
- （任意）ONNX Runtime: モデル推論高速化/配布簡略化

### 8.3 幾何/最適化

- NumPy: 数値処理
- Shapely: 当たり判定、交差面積計算
- OR-Tools CP-SAT（任意）: 多吹き出し時の配置最適化

### 8.4 UI/実行

- CLI: バッチ・自動処理用
- FastAPI + Gradio: ローカルGUI（確認・微調整）

## 9. 推奨アーキテクチャ

`src/speechbubble/` 配下を以下分割:

- `io/` 入出力、設定、メタデータ保存
- `vision/` 顔検出、ランドマーク、サリエンシー/マスク
- `layout/` 候補生成、スコアリング、最適化
- `render/` 吹き出し図形、しっぽ、テキスト描画
- `app/` CLI・Web API

処理パイプライン:

1. 画像前処理
2. 顔/ランドマーク推定
3. 配置候補生成
4. コスト最小化で候補採択
5. 吹き出し+文字を描画
6. 画像/JSONを書き出し

## 10. データ戦略

- 初期: 手元の作例で評価データセットを作成（50〜200枚）
- 中期: 失敗ケースを継続収集して回帰テスト化
- 学習導入時: アノテーション方針を先に固定（顔、吹き出し、しっぽ先端）
- 公開データ利用時は利用規約を厳守（特に商用可否）

## 11. リスクと対策

- R-001: 顔が小さい/横顔/遮蔽で検出不安定
  - 対策: 検出器二段構え + フォールバック配置
- R-002: 背景が密で重なり回避が難しい
  - 対策: サリエンシーマップ導入と複数候補探索
- R-003: 日本語組版で文字溢れ
  - 対策: 描画前に文字計測し、自動改行と最小フォント調整
- R-004: ライセンス不整合
  - 対策: `THIRD_PARTY_LICENSES.md` を初期から運用

## 12. 開発ロードマップ（推奨）

### Phase 0: 設計固め（1〜2週間）

- 要件確定
- 評価データ作成
- コスト関数定義

### Phase 1: MVP実装（2〜4週間）

- 顔検出 + ルール配置 + 描画
- CLI + JSON出力
- 最低限の回帰テスト

### Phase 2: 品質改善（2〜4週間）

- サリエンシー/マスク導入
- 複数吹き出し最適化
- Web UI追加

### Phase 3: 高度化（継続）

- 学習モデルでスコア関数を改善
- 縦書き/ページ全体最適化
- バッチ処理強化

## 13. 初期実装の意思決定（この時点の提案）

- 採用: Python + OpenCV + MediaPipe + Pillow + FastAPI/Gradio
- 保留: SAM 2 などの重い分割モデル（MVP後に必要性で判断）
- 採用: ルールベース最適化を先に完成
- 採用: JSONメタデータ出力を早期導入（編集連携のため）
